# 知识图谱服务部署问题解决总结

## 问题描述

在将索克生活知识图谱服务部署到Kubernetes集群时，遇到了以下问题：

1. Pod一直处于CrashLoopBackOff或ImagePullBackOff状态
2. 容器启动后出现"exec format error"错误
3. 镜像无法从私有仓库拉取

## 根本原因分析

经过调查，发现了以下根本原因：

1. **架构不匹配**：本地开发环境使用M1 Mac（ARM64架构）构建的镜像，而Kubernetes集群节点是AMD64架构，导致二进制不兼容，出现"exec format error"。

2. **镜像仓库配置问题**：Kubernetes集群没有正确配置阿里云私有镜像仓库的访问凭证，导致ImagePullBackOff错误。

3. **Dockerfile配置不当**：原Dockerfile没有明确指定目标架构，导致默认使用构建机器的架构（ARM64）。

4. **部署配置冲突**：Helm charts中的nodeSelector配置存在冲突，影响了Pod的调度。

## 解决方案

### 1. 架构兼容性解决方案

创建了针对AMD64架构优化的Dockerfile和构建脚本：

- **Dockerfile.amd64**：明确指定`--platform=linux/amd64`参数，确保即使在ARM64机器上构建也能生成AMD64兼容的镜像。
- **build-amd64-push-public.sh**：专门用于构建AMD64镜像并推送到公共仓库。

### 2. 镜像仓库访问解决方案

- 创建了针对公共仓库的构建脚本，避免私有仓库认证问题。
- 更新部署配置，使用更可靠的公共仓库镜像。
- 提供了镜像仓库Secret创建指南。

### 3. 部署配置优化

- 修复了values.yaml中的nodeSelector配置，移除了冲突设置。
- 更新了容器端口配置，统一使用8080和9464端口。
- 优化了健康检查配置，延长初始等待时间。

### 4. 自动化工具

创建了一系列实用脚本简化操作：

- **docker-build-amd64.sh**：快速构建AMD64镜像。
- **build-amd64-push-public.sh**：构建并推送到公共仓库。
- **update-k8s-deployment.sh**：快速更新Kubernetes部署。

## 最终效果

1. 成功解决了架构不兼容问题，保证了在AMD64集群上的正常运行。
2. 建立了可靠的镜像构建和推送流程，支持多架构。
3. 优化了部署配置，提高了部署的成功率和稳定性。
4. 创建了一套自动化工具，简化了日常维护操作。

## 经验教训

1. **始终明确指定目标架构**：在Dockerfile中使用`--platform`参数明确指定目标架构，避免默认使用构建机器架构。
2. **使用多阶段构建**：确保每个构建阶段都指定正确的平台参数。
3. **采用标准化标签**：使用包含架构信息的镜像标签（如1.0.0-amd64）。
4. **完善部署文档**：创建详细的部署指南，包括故障排除步骤。
