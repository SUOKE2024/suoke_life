# Agent Coordinator Service Kubernetes 部署架构

本文档描述了 Agent Coordinator Service 在 Kubernetes 集群中的部署架构设计和实施细节。

## 架构概览

Agent Coordinator Service 采用多层架构部署在 Kubernetes 集群中，确保高可用性、可扩展性和安全性。

## 架构组件

### 1. 核心服务组件

- **部署 (Deployment)**: 定义 Agent Coordinator Service 的部署规格
  - 副本数: 生产环境 2+, 开发环境 1
  - 镜像: suoke-registry.cn-hangzhou.cr.aliyuncs.com/suoke/agent-coordinator
  - 健康检查: 存活探针和就绪探针
  
- **服务 (Service)**: 暴露 Agent Coordinator Service
  - 类型: ClusterIP
  - 端口: 80 -> 8080

### 2. 配置管理

- **ConfigMap**: 存储服务配置
  - 服务器参数配置
  - 微服务连接信息
  - 日志级别和格式

### 3. 可扩展性组件

- **水平 Pod 自动缩放器 (HPA)**: 根据 CPU 和内存使用率自动调整副本数
  - 最小副本数: 2
  - 最大副本数: 5
  - 目标 CPU 利用率: 70%
  - 目标内存利用率: 80%

### 4. 可用性保障

- **节点选择器**: 确保服务部署在适当的节点上
  - 架构: AMD64
  
- **Pod 中断预算 (PDB)**: 确保维护期间服务不中断
  - 最小可用: 1

### 5. 数据持久化

- **持久卷声明 (PVC)**: 为服务提供持久化存储
  - 大小: 5Gi
  - 类型: 标准存储
  - 挂载路径: /app/data

### 6. 网络安全

- **网络策略**: 控制 Pod 的进/出流量
  - 入站: 允许 API Gateway 和系统服务访问
  - 出站: 允许访问依赖服务和 DNS

## 部署流程

### CI/CD 集成

服务通过 GitLab CI/CD 管道自动构建和部署：

1. **构建阶段**: 编译代码并构建 Docker 镜像
2. **测试阶段**: 运行单元测试和集成测试
3. **发布阶段**: 推送 Docker 镜像到仓库
4. **部署阶段**: 使用 Kustomize 更新 Kubernetes 配置并应用

### ArgoCD 集成

生产环境部署通过 ArgoCD 进行管理：

1. ArgoCD 监控 Git 仓库中的 Kubernetes 配置变更
2. 变更检测后自动同步到 Kubernetes 集群
3. 支持回滚和版本历史管理

## 监控与可观测性

### 健康检查

- **存活探针**: `/health` 端点，确保服务正在运行
- **就绪探针**: `/health` 端点，确保服务可以接收流量

### 日志收集

所有日志流被收集到中央日志系统（EFK Stack）：

- **Elasticsearch**: 存储和索引日志
- **Fluentd**: 收集和传输日志
- **Kibana**: 日志可视化和分析

### 指标监控

Pod 暴露的指标由 Prometheus 收集，并通过 Grafana 进行可视化：

- CPU 和内存使用率
- 请求延迟和错误率
- 业务指标（会话数、任务完成率等）

## 资源管理

### 资源需求和限制

- **生产环境**:
  - 请求: CPU 500m, 内存 512Mi
  - 限制: CPU 1, 内存 1Gi

- **开发环境**:
  - 请求: CPU 200m, 内存 256Mi
  - 限制: CPU 500m, 内存 512Mi

## 安全考量

1. **镜像安全**: 使用基于 Alpine 的小型基础镜像，减少攻击面
2. **网络安全**: 实施网络策略，限制 Pod 间通信
3. **运行时安全**: 以非特权用户身份运行服务
4. **配置安全**: 敏感配置通过 Secret 管理

## 灾难恢复

1. **数据备份**: PVC 中的数据定期备份
2. **多区域部署**: 关键生产工作负载分布在多个可用区
3. **恢复流程**: 记录在 DR 手册中的详细恢复步骤
