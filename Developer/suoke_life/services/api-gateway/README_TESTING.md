# API网关服务测试说明

## 测试框架概述

API网关服务的测试框架采用多层次测试策略，包括单元测试、集成测试和性能测试。主要使用Go语言的标准测试库和第三方测试工具。

## 测试类型

### 1. 单元测试

单元测试主要针对每个组件的独立功能，通过模拟外部依赖来验证组件的正确性。API网关服务的单元测试主要涵盖：

- **中间件测试**：认证、限流、缓存等中间件
- **代理功能测试**：请求转发和重试逻辑
- **服务发现测试**：服务注册、发现和心跳机制
- **配置管理测试**：配置加载和解析

### 2. 集成测试

集成测试验证多个组件协同工作的正确性，重点关注组件间的交互。主要包括：

- API网关与认证服务的集成
- API网关与后端服务的代理功能
- 中间件链的协同工作

### 3. 性能测试

性能测试评估系统在高负载下的表现，包括：

- 响应时间测试
- 吞吐量测试
- 资源使用测试
- 限流效果测试

## 测试用例设计原则

1. **独立性**：每个测试用例应该相互独立，不依赖于其他测试的执行结果。

2. **全面性**：测试应覆盖正常流程和各种边缘情况，包括错误处理和容错机制。

3. **自动化**：所有测试应能自动执行，无需人工干预。

4. **可复现**：测试结果应是确定性的，避免随机性和外部依赖。

5. **性能敏感**：性能测试应设置明确的基准和阈值，便于持续监控。

## 测试实施

### 运行所有测试

使用提供的测试脚本运行所有测试：

```bash
./scripts/run_tests.sh
```

### 运行特定测试

运行单个模块的测试：

```bash
go test -v ./internal/middleware
go test -v ./internal/proxy
go test -v ./internal/discovery
```

运行特定测试用例：

```bash
go test -v ./internal/middleware -run TestAuth
go test -v ./internal/proxy -run TestProxy_RetryMechanism
```

### 生成测试覆盖率报告

```bash
go test -coverprofile=coverage/coverage.out ./internal/...
go tool cover -html=coverage/coverage.out -o coverage/coverage.html
```

## 模拟服务

为了便于测试，我们提供了多个模拟服务：

1. **MockLogger**：用于测试的日志记录接口实现
2. **MockBackendServer**：模拟后端服务的HTTP响应
3. **MockDiscoveryService**：模拟服务发现注册中心

## 测试数据管理

测试数据应遵循以下原则：

1. 测试数据应尽可能简单，只包含验证功能所需的最小数据集。
2. 敏感测试数据（如测试令牌）不应硬编码在测试文件中，而应通过环境变量或配置文件引入。
3. 模拟后端服务的响应数据应与实际服务保持一致的格式。

## 持续集成

在CI/CD流程中，测试通常由GitLab CI自动执行：

1. 每次提交代码后自动运行单元测试
2. 合并到主分支前运行完整测试套件
3. 定期运行性能测试，监控性能变化

## 问题排查

如果测试失败，请遵循以下步骤：

1. 检查测试环境和依赖是否正确配置
2. 使用详细模式运行测试，查看具体失败原因：`go test -v`
3. 检查测试用例中的模拟服务配置是否正确
4. 查看项目日志，了解更多细节

## 测试维护

为保持测试的有效性，需要定期进行以下工作：

1. 更新测试用例以适应新功能和API变化
2. 移除过时的测试用例
3. 优化性能测试基准
4. 调整模拟服务行为以匹配实际服务的变化
