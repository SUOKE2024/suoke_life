# 索儿服务日常活动模块测试文档

本文档介绍索儿服务日常活动模块的测试策略、测试类型和运行方法。

## 测试策略

日常活动模块的测试遵循"测试金字塔"原则，包含以下几个层次：

1. **单元测试**：测试各个组件的功能，确保它们独立正常工作
2. **边界条件测试**：测试各种异常情况和边界条件，确保系统健壮性
3. **集成测试**：测试组件间的交互，确保它们可以协同工作
4. **端到端测试**：测试完整的用户流程，模拟真实用户行为
5. **性能测试**：测试系统在各种负载下的性能表现

## 测试类型详情

### 单元测试

单元测试主要覆盖以下组件：

- **模型**：测试数据格式转换和验证逻辑
- **服务**：测试业务逻辑（依赖被模拟）
- **控制器**：测试API处理逻辑（依赖被模拟）
- **数据库服务**：测试数据库操作（实际数据库被模拟）

单元测试文件位于 `/tests/unit/` 目录。

### 边界条件测试

边界条件测试主要关注异常情况和边界场景：

- **无效参数**：测试系统对空值、非法值的处理
- **极端值**：测试系统对极大值、极长字符串等的处理
- **异常处理**：测试系统对数据库错误等异常的处理
- **错误转换**：测试错误类型的正确转换和传播

边界条件测试文件位于 `/tests/unit/` 目录，文件名包含 `.boundary.`。

### 集成测试

集成测试主要测试以下方面：

- **数据库集成**：测试与实际数据库的交互
- **模块集成**：测试多个组件合作时的行为
- **API路由集成**：测试HTTP端点与控制器的集成

集成测试需要实际数据库连接，文件位于 `/tests/integration/` 目录。

### 端到端测试

端到端测试模拟完整的用户场景：

- **用户流程**：测试完整的用户操作流程
- **错误场景**：测试用户操作中可能出现的错误

端到端测试需要完整的运行环境，文件位于 `/tests/e2e/` 目录。

### 性能测试

性能测试评估系统在各种条件下的性能：

- **响应时间**：测试各API端点的响应速度
- **大数据量**：测试处理大量数据时的性能
- **并发处理**：测试并发请求时的性能

性能测试默认被禁用，需要通过环境变量启用，文件位于 `/tests/performance/` 目录。

## 运行测试

### 使用测试脚本

项目提供了便捷的测试脚本，可以运行各类测试：

```bash
# 运行测试脚本
./scripts/run-tests.sh
```

脚本提供以下选项：

1. **基础单元测试**：运行所有基本单元测试
2. **边界条件测试**：运行边界条件测试
3. **集成测试**：运行数据库集成测试
4. **端到端测试**：运行完整用户流程测试
5. **性能测试**：运行性能评估测试
6. **所有测试**：按顺序运行上述所有测试
7. **生成覆盖率报告**：生成测试覆盖率报告

### 手动运行测试

也可以使用Jest命令直接运行测试：

```bash
# 运行所有单元测试
npm test -- --testPathPattern=tests/unit

# 运行边界条件测试
npm test -- --testPathPattern=boundary

# 运行集成测试
npm test -- --testPathPattern=tests/integration

# 运行端到端测试
npm test -- --testPathPattern=tests/e2e

# 运行性能测试
RUN_PERF_TESTS=true npm test -- --testPathPattern=tests/performance
```

### 生成测试覆盖率报告

```bash
# 生成覆盖率报告
npm test -- --coverage
```

覆盖率报告将生成在 `coverage/` 目录下。

## 数据库测试配置

集成测试、端到端测试和性能测试需要连接MySQL数据库。测试使用以下环境变量配置数据库连接：

- `TEST_DB_HOST`：数据库主机（默认：localhost）
- `TEST_DB_PORT`：数据库端口（默认：3306）
- `TEST_DB_USER`：数据库用户名（默认：test）
- `TEST_DB_PASSWORD`：数据库密码（默认：test）
- `TEST_DB_NAME`：数据库名称（默认：test_db）

您可以通过环境变量自定义这些设置：

```bash
# 使用自定义数据库运行集成测试
TEST_DB_HOST=my-db-server TEST_DB_USER=myuser TEST_DB_PASSWORD=mypass TEST_DB_NAME=test_db npm test -- --testPathPattern=tests/integration
```

## 测试数据

测试会自动创建并清理测试数据，对于集成测试、端到端测试和性能测试，每个测试会话都会使用唯一的用户ID，以避免测试之间的干扰。

每个测试会话结束时，测试脚本会尝试清理所有创建的测试数据。如果测试异常终止而未能清理数据，可以通过以下SQL手动清理：

```sql
DELETE FROM user_daily_activities WHERE user_id LIKE 'test-%';
DELETE FROM user_activity_metrics WHERE user_id LIKE 'test-%';
DELETE FROM user_activity_goals WHERE user_id LIKE 'test-%';
```

## 持续集成

此测试套件已配置为在持续集成环境中运行。在CI环境中，系统会自动运行单元测试和边界条件测试，但不会运行需要数据库的测试，除非CI环境已配置了适当的数据库。

CI配置文件位于 `/.github/workflows/test.yml`。

## 测试覆盖率目标

项目的测试覆盖率目标如下：

- **语句覆盖率**：>90%
- **分支覆盖率**：>85%
- **函数覆盖率**：>95%
- **行覆盖率**：>90%

当前测试覆盖率可通过运行覆盖率报告查看。
